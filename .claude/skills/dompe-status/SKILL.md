# Dompe CEO Status Report Generator

Generate the weekly CEO status update .docx for the Dompe Document Authoring Hub project.

## Workflow

### Step 1: Locate the week folder

Source data lives in `~/projects/dompe-crd-srd-auth/nat-update/<week-folder>/`. The user will specify or reference which week folder to use. Each folder contains:
- `<version>.md` files — release notes for versions shipped that week
- `Eval-scores.md` — evaluation score table with clinical questions, quality scores, total times

Read **all** files in the week folder to gather content.

### Step 2: Read the previous week's report

Find the most recent prior week folder in `~/projects/dompe-crd-srd-auth/nat-update/` and read its `drafts/status-2.md` (or the config JSON if one exists). This provides:
- The previous RAMP plan state (week number, milestone statuses)
- Previous eval scores (carry forward; new evals are appended)
- Blocker status (check if prior blockers are now resolved)

### Step 3: Compose the config JSON

Create a `config.json` in the week folder. See [references/report-format.md](references/report-format.md) for the full schema and all fields.

Key decisions to make (ask the user if unclear):
- **Week number**: Increment from prior week
- **RAMP status**: Usually stays "risk" until scores hit 80%+
- **Eval handling**: Include new evals from `Eval-scores.md`, carry forward prior evals. If scores are invalidated (environment issues), show with a caveat footnote.
- **Blockers**: Mark prior blockers as resolved if fixes shipped. Add new blockers.
- **Decision point date**: Typically the Monday after the report date.

Writing style for summary bullets:
- Direct, factual, data-driven
- Lead with schedule status and week number
- Highlight resolved blockers with `**RESOLVED:**` prefix
- Call out major features shipped
- Note quality gap with specific numbers
- No embellishment, no promotional language

### Step 4: Generate the .docx

```bash
NODE_PATH=$(npm root -g) node <this-skill-dir>/scripts/generate_report.js <week-folder>/config.json
```

Output: `CEO_Status_Update_<date>.docx` in the week folder.

### Step 5: Verify

```bash
cd <week-folder>
soffice --headless --convert-to pdf CEO_Status_Update_*.docx
pdftoppm -jpeg -r 150 CEO_Status_Update_*.pdf preview
```

Read the preview image to confirm: single-page layout, all sections present, color-coded tables render properly, no overflow.

### Step 6: Save a draft copy

Save the markdown-formatted draft as `drafts/status-2.md` in the week folder for future reference.

## Key Constraints

- **Single page**: Must fit on one US Letter page. Reduce content if it overflows.
- **No fabrication**: Never invent dates, scores, or version numbers. Leave blank if unknown.
- **Carry forward evals**: Always include all prior evals in the table, not just new ones.
- **Tone**: Executive-focused, plain English, metrics-heavy. No AI-sounding language.
- **Stakeholders**: To Nathalie Dompe (CEO), CC: Neal Patel, Ivy Chang, Janice Lin, Andrew Sheldon, Beth Scott, Jonathan Yu, Alexander Downey, Sina Sojoodi.

## Stakeholder Glossary

When distributing the report to legal counsel or other external non-technical stakeholders, include the following plain-English glossary. Append it as a companion note in the email body or as a second page in the document.

**Before sending the glossary, verify each definition against source files.** See Glossary Verification Protocol below.

| Term | Definition |
|------|------------|
| **SRD / No Data SRD** | Standard Response Document — a structured document that Dompé's Medical Information team sends to healthcare professionals in response to clinical inquiries about their products. A "No Data SRD" is a variant used when no clinical study data applies to the topic section. |
| **RAMP Plan** | The 12-week Rollout Adoption Maturity Plan for transitioning from manual to AI-assisted document authoring, with milestones at Week 1 (training), Week 4 (quality baseline review), Week 8 (quality checkpoint), and Week 12 (full production). |
| **Evals / Evaluations** | Structured quality tests where a document is generated by the tool and scored against a 26-point rubric measuring output quality vs. the expected clinical standard. |
| **Environment limitations** | Configuration or infrastructure gaps in the test setup that caused test failures unrelated to the tool's functionality (e.g., an undocumented scheduled task that wasn't running during testing). Only describe the specific cause if it is explicitly stated in the source files for that week. |
| **Veeva** | Veeva MedComms — a cloud-based document management system used by Dompé for its Medical Legal Review compliance workflow. Once the tool generates a completed document, it is uploaded to Veeva MedComms for review by the Medical Review Committee before being sent to the healthcare professional. |
| **Cron job** | An automated scheduled task that runs at set intervals without manual intervention. A missing or undocumented cron job means the automation wasn't running during testing. |
| **URL-based sessions** | Document authoring sessions now have permanent web addresses (URLs), allowing users to bookmark or share a direct link to a specific document in progress. |
| **Tabbed PDF workspace** | An interface that lets users open multiple reference documents in separate tabs within the tool, with drag-and-drop reordering — similar to browser tabs. |
| **AI stages** | The portion of the document authoring workflow handled by the AI (automated search, summarization, and draft generation), as distinct from the manual review and editing stages. |

## Glossary Verification Protocol

When preparing or updating glossary definitions for external distribution, verify each definition against source files before sending. Do not rely on memory or prior drafts.

### Source files to check

| What to verify | Where to look |
|----------------|---------------|
| SRD / CRD full names and purpose | `prds/dompe-crd-srd-auth-V5_prd_version_16.md` — Introduction section |
| RAMP Plan duration and milestones | `improvement/CRD_SRD_RAMP_Plan.md` |
| Eval scoring rubric (point total) | `nat-update/<week>/Eval-scores.md` — score format (e.g., `17/26`) |
| Env issue / environment limitations cause | `nat-update/<week>/Eval-scores.md` — Explanation section. **Only state the specific cause if it is written there. If it says "explained in chat" or similar, leave the cause unspecified.** |
| Veeva product name and use | `prds/dompe-crd-srd-auth-V5_prd_version_16.md` — Document Finalization section |
| Feature descriptions (URL sessions, tabs, import bug) | `nat-update/<week>/<version>.md` — release notes for that week |

### Rules

- **Never fabricate specifics.** If a cause, number, or detail is not in the source files, omit it rather than infer it.
- **Quote before interpreting.** Find the exact text in the source file first, then write the plain-English definition from that.
- **Check product names exactly.** Veeva MedComms, not Veeva Vault. Standard Response Document, not Scientific Research Document.
- **Numbers must match source.** RAMP is 12 weeks. Rubric is 26 points. Do not round or approximate.
